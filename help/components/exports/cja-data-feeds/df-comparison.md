# Compare data feeds in Customer Journey Analytics and Adobe Analytics

Data feeds in both Customer Journey Analytics and Adobe Analytics allow you to export raw data to third-party platforms. If you previously used data feeds in Adobe Analytics, use the following information to understand differences in available features and concepts:

| **Concepts and configuration options** | **Customer Journey Analytics** | **Adobe Analytics** |
|---------|----------|---------|
| **Data input**<br/>The type of data that can be collected and included in data feeds. |Supports cross-channel data input, including web data, call center data, point-of-sale data, and more.  | Supports web and mobile data input only. |
| **Data processing**<br/> Data often needs to be processed before it is useful for reporting, and this processing affects data feeds functionality. Data is processed at different stages, depending whether you are using Customer Journey Analytics or Adobe Analytics. | Data is processed at report time, and therefore many reporting features in Customer Journey Analytics can be used to change historical data, such as stitching, derived fields, Data Prep, segmentation, and the reporting window.  | Data is processed at collection time, and therefore reporting features such as processing rules, VISTA rules, stitching, and segmentation do not affect historical data.  |
| **Delivery frequency**<br/>  Determines:<ul><li>How often the data feed is sent</li><li>The window of time included in the data feed<p>In other words, after the frequency window ends and the data feed is sent, a new frequency window begins.</p></li></ul> |  The following options are available:<ul><li>**Daily**: Feeds contain a full day's worth of data, from midnight to midnight in the data view's time zone. Use this option for backfill feeds or live feeds.</li><li>**Hourly**: Feeds contain a single hour's worth of data. Use this option for live feeds.</li></ul>  <p>Late-arriving hits are configurable based on the [!UICONTROL **Processing delay**]. <p>Example: 1 hour</p><p>For more information, see the "Late-arriving hits" explanation below.</p> | The following options are available:</p><ul><li>**Daily**: Feeds contain a full day's worth of data, from midnight to midnight in the report suite's time zone. Use this option for backfill or historical data, or for continuing feeds.</li><li>**Hourly**: Feeds contain a single hour's worth of data. Use this option for continuing feeds.</li></ul>  In addition, 15-minute feeds are possible, but are not available by default. </p><p>Late-arriving hits are configurable based on the **[!UICONTROL Lookback window]**.</p><p>Example: 1 hour</p><p>For more information, see the "Late-arriving hits" explanation below.</p> |
| **Late-arriving hits**<br/> Late-arriving hits are hits that arrive during the current frequency window but have a timestamp that falls in a previous frequency window (for example, via timestamped hits or data sources). <p>When hits arrive late, they are batched into the next data feed file that is sent, even though their timestamps belong to an earlier window.</p>| Hits are automatically reordered by timestamp in the output (for example, late hits can appear before on-time hits if their timestamps are earlier). There is no change feed, so original values are persisted.<p>The **[!UICONTROL Processing delay]** configuration option determines the cutoff for late or out-of-order hits that are delivered outside of the **[!UICONTROL Delivery frequency]** that is selected (such as **[!UICONTROL Hour]** or **[!UICONTROL Day]**). <p>Visits must start after this cutoff in order to be included; visits that start before the cutoff and end within the processing delay are not included.</p> <p>Required for sessions, persistence, and segments.</p><p>Not used for dimensions. Dimensions are controlled per dimension based on the dimension's allocation and expiration. Dimension lookbacks cannot exceed the processing delay.</p>  | Hits are shown in the order in which they are received; they are not reordered according to timestamp. <p>Late-arriving hits can be included or excluded, and are configurable with the **[!UICONTROL Late-arriving hits]** configuration option, which is available for this specific purpose.</p> <p>If late-arriving hits are included, the **Lookback window** configuration option determines the cutoff for late or out-of-order hits that are delivered outside of the **[!UICONTROL Delivery frequency]** that is selected (such as **[!UICONTROL Hour]** or **[!UICONTROL Day]**).</p><p>Visits must start after this cutoff in order to be included; visits that start before the cutoff and end within the lookback window are not included.</p><p>Not used for persistence, sessions, or dimensions (these are included in the raw data that is collected). </p> |
| **Out-of-order hits**<br/> Out-of-order hits are hits whose timestamps don't match the order in which they were received. | Performs report-time processing on out-of-order hits and reorders data by timestamp, per person. <p>You can configure a processing delay that controls how long the system waits before triggering the export, after the frequency window closes. Data that arrives after the processing delay is included in the next delivery.</p><p>**Important**: The ultimate consumer of your data feed data must be able to handle timestamps that are out of order, per person, because data feed data might include late or out-of-order hits with timestamps earlier than the frequency window.</p> | Data must arrive in order, per visitor. Out-of-order arrival causes processing issues. |
| **Backfill window**<br/>Exports historical data between two past dates.  | The backfill limit cannot exceed the rolling data window of the connection. The rolling data window of a connection depends on the Customer Journey Analytics plan: <ul><li>**Select**: 13 months</li><li>**Prime and Ultimate**: 25 months</li></ul><p>Backfill is available only when creating a backfill feed (not a live feed).</p> | The backfill limit cannot exceed the report suite data retention limit, which is 25 months by default. |
| **Segmentation**<br/>   | Segmets that you configure in Customer Journey Analytics can be applied to data feeds or to the data view.<p>(Person, Session, Event) Person containers use the entire reporting window; can expand DF hit output based on segment logic.</p> | Segments cannot be applied. |
| **Stitching**<br/> <!--could be included in the data processing section instead--> | Stitching can be used to join multiple datasets (to support cross-channel analysis). | Stitching is not supported. |
| **Schema**<br/> The schema determines which columns are available to include in a data feed. <p>For more information, see [Schema architecture](/help/components/exports/cja-data-feeds/df-column-mapping-prep.md#schema-architecture).<!-- could be included in the data processing section instead --></p> | Uses an XDM schema. XDM schemas include structured, hierarchical fields, potentially containing multiple dimensions and metrics within a single column (arrays; arrays within arrays). <p>In addition to fields included in the XDM schema, Customer Journey Analytics also allows for the creation of derived fields, allowing users to create fields within Customer Journey Analytics which can then be included in data feed output.</p> | Uses a product list with a long string of proprietary, semicolon-delimited fields.|
| **Lookups**<br/> <!-- (could be included in the data processing section instead)-->Dynamic lookups allow you to receive additional lookup files in your data feed otherwise not available. | Not needed, because lookups and classifications are both available as dimensions curated directly in the data view. When you curate a lookup or classification as a dimension in the data view, the resolved values appear as regular columns in the Parquet output, inline with the event data, not as separate reference files. | Used to match a number from a data feed column to an actual value. Specific to a certain set of things (Browser, OS, Mobile device, and they're applied as a separate file that comes with the data feed.)|
| **Session definition**<br/> <!--(could be included in the data processing section instead)--> | Defined in the data view .| Defined at collection time. |
| **Calculated metrics**<br/>   | Not available | Not available |
| **Persistence model**<br/>  <!--(could be included in the data processing section instead)--> | Flexible; based on frequency window and processing delay. Only "Last Touch" (Most Recent) and "First Touch" (Original).<p>Supports all the dimension allocation settings that are available in a data view: Original, Most Recent, All, First Known, and Last Known. (For more information about these allocation options, see [Persistence component settings](/help/data-views/component-settings/persistence.md).)</p> | |
| **Output file format**<br/> The format in which data feed data is delivered. | Parquet <p>Supports complex nested and structured data natively and is accepted by modern data warehouses. This is an important enhancement when exporting information like product lists: in Adobe Analytics, `product_list` is a proprietary delimited string that can be parsed only with custom logic. In Customer Journey Analytics, the equivalent data is represented as structured objects within Parquet (arrays and nested structures).<p>Requires a Parquet-aware tool or library to read, such as Apache Arrow, Spark, pandas, BigQuery, or Snowflake.</p>| TSV files <p>Flat, simple rows of human-readable text.</p>  |
| **Delivery destinations**<br/> The destinations where data feed data can be sent. | Cloud destinations, including Amazon S3, Azure RBAC, Azure SAS, and Google Cloud Platform. | Cloud destinations, including Amazon S3, Azure RBAC, Azure SAS, and Google Cloud Platform.<p>Also supports SFTP.</p>|
